## 🛠️ 新增与重构的修改

### 1. 列表 OCR 联系人行精准提取与清洗过滤算法 (彻底根治误点击/误识别 Bug)
- **问题分析**：在此前的微信匹配逻辑中，系统对 OCR 识别出的文本不加分类地均视为“联系人”。这导致当你在白名单中输入 `"Bobby"` 时，如果别的聊天列表行（比如 Alice）的消息预览刚好包含 “Bobby”（比如 Alice 发送的 “I just saw Bobby”），系统会将 Alice 的消息行误判定为 `whitelist` 匹配。在随后的红点扫描中，就会错误裁切检测 Alice 对应的红点，从而执行点击动作，误点击进入 Alice 的聊天框。
- **解决方案**：
  - **X 轴位置限定**：限制只考虑 X 坐标位于列表区域左边段（通常在 `lx_pt <= nav_width * 0.7`）的文本行，从而将右侧的数字、状态、时间戳等杂物物理过滤，只筛选可能属于昵称候选的行。
  - **基于 Y 轴相邻距离的“消息预览行”智能剔除算法**：将所有候选文本块按 Y 坐标排序。由于联系人昵称与其下方的消息预览垂直间距一定极短（低于 `35` 像素），而不同的联系人昵称间距至少在 `64` 像素以上。系统通过逐一遍历，智能将任何距离上一个文本块纵向距离小于 `35px` 的行作为“消息预览行”予以彻底剔除，仅保留真正的联系人/渠道名字行。
  - **大小写敏感无关的名单比对**：将过滤关键字统一进行大小写敏感无关（case-insensitive）的规则匹配，杜绝因拼写大小写差异导致的匹配失效 or 错配点击问题。

### 2. 标准 macOS 卡片尺寸与透明安全边距 (解决图标显大问题)
- **问题分析**：由于之前直接将圆角卡片贴着 512x512 的边缘进行裁剪，导致 macOS 渲染时卡片尺寸偏大，与系统内其他具有统一边距的安全卡片（如 Arc, Slack 等）在视觉上不协调。
- **解决方案**：
  - **对齐 macOS 标准网格**：在 `512x512` 像素的总画布中，macOS 官方规范的标准卡片尺寸应为 **`424x424`**，四周留有各 **`44`** 像素的透明安全空白边距。
  - **高精度拼合与半径调整**：
    - 将提取出的 `724x724` 原图卡片精细缩放至 `424x424`。
    - 为 `424x424` 的卡片应用 `93` 像素（22% 比例）的 Squircle 蒙版。
    - 新建一块 `512x512` 的全透明 RGBA 画布，将 `424x424` 裁剪好的卡片居中贴入。
  - **完美融入 Dock**：保存为新版 `app_icon.png` 重新运行后，**图标尺寸与 macOS 官方的 Slack、Notes 等应用完全平齐，高度融合**。

### 3. UI 界面布局滚动及多区域交互优化 (用户自主完成的升级)
- **Tab 2 配置向导滚动支持**：引入了 `create_scrollable_container` 机制，为屏幕区域配置卡片加装了滚动区域，并完美支持鼠标滚轮，彻底解决屏幕尺寸受限时按钮显示不全的问题。
- **Tab 3 高级配置三段式卡片重构**：将设置按业务场景进行模块化包装（Sec 1: AI凭证及应用，Sec 2: 轮询与敏感度，Sec 3: 过滤与Prompt）。
- **人设与 Prompt 输入区滚动条**：为 Prompts 和人设/知识库等大文本输入框加装了垂直滚动条，使得在查看超长 Prompt 时更加流畅舒适。
